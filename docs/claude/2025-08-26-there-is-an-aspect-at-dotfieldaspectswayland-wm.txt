● Summary: User Aspect Import Issue Analysis

  Problem

  The user-specific wayland-wm aspect (containing Nemo
   file manager config) isn't being imported when
  hosts use the sway aspect, even though sway.nixos
  imports wayland-wm.nixos via standard NixOS module
  imports.

  Root Cause

  In src/modules/dotfield/nixos.nix:49-51, user
  aspects are only imported when they match aspects
  explicitly listed in the host configuration:

  mirroredAspects = userConfig.aspects
    |> (map (v: customAspects.${v._name} or { }))
    |> filter (v: v != { });

  Since hosts list sway but not wayland-wm, the user's
   wayland-wm aspect never gets imported.

  Key Architecture Understanding

  - Features: Named by filename (e.g., nemo.nix,
  mpv.nix) in src/features/ and
  src/users/cdom/features/
  - Aspects: Logical groupings that features
  contribute to (e.g., wayland-wm, workstation)
  - Dependencies: Currently handled via standard NixOS
   module imports (e.g., sway.nixos imports
  wayland-wm.nixos)
  - Global aspects: Output to
  flake.modules.(nixos|homeManager).<aspect-name>,
  accessed via
  dotfield.aspects.<aspect-name>.(nixos|home)
  - User aspects: Internal-only, no flake-parts
  output, same semantics otherwise

  Solution Options Considered

  1. Static Analysis - Traverse imports at build time
  (too "magical")
  2. Explicit Dependencies - Add depends field to
  aspects (interesting but major change)
  3. Manual Lists - Require explicit user aspect lists
   (too much maintenance burden)
  4. Hybrid Discovery - Extract dependencies from
  imports with explicit validation

  Next Steps

  Need to decide on approach that balances usability,
  maintainability, and predictability without
  excessive "magic."
